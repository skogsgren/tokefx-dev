Large language models (LLMs) do not process text as sequences of words. Instead, their input is segmented into tokens, many of which correspond only to fragments of words. For example, the city name \textit{Paterson} is represented in the GPT-2 tokenizer as \texttt{['Pat', 'erson']}. Because all subsequent computation in the model operates over token representations, any information that spans multiple tokens must be reconstructed from context during inference. Understanding how models perform this reconstruction is therefore important for explaining their robustness to typos \citep{zhao2026evaluatingrobustnesslargelanguage, cao2023} and non-canonical segmentation \citep{gutierrez2023, zheng2026}. To examine this issue, we need to consider how tokenization shapes model input, how this input is transformed within the model, and how prior work has analyzed the internal computations of modern LLMs.

This chapter is structured as follows: Section~\ref{sec:background_tokenization} introduces common tokenization algorithms, Section~\ref{sec:background_transformer} outlines the relevant components of the transformer architecture, and Section~\ref{sec:background_interpretability} presents work in mechanistic interpretability, including the \textit{detokenization hypothesis}, which proposes that early layers reconstruct larger linguistic units from fragmented inputs.

\section{Tokenization in LLMs}\label{sec:background_tokenization}

Compression-based subword tokenization algorithms are the standard for modern LLMs \citep[Chapter 2]{jm3}. The primary motivation for subword tokenization is to mitigate the out-of-vocabulary problems of earlier word-based approaches \citep{mielke2021}. In practice, several related approaches are used to construct such subword vocabularies, differing primarily in how they segment text and assign probabilities to candidate units. The most widely used methods include \textbf{Byte Pair Encoding (BPE)}, \textbf{Unigram} language model tokenization, and \textbf{byte-level} tokenization. These approaches represent different trade-offs.

\begin{description}

\item[Byte-Pair Encoding]
The Byte-Pair Encoding (BPE) algorithm, introduced by \citet{bpe2}, is a text compression algorithm. It was applied to Neural Machine Translation (NMT) by \citet{bpe}, and was introduced to solve out-of-vocabulary issues. Compared to earlier approaches which relied on traditional word-mappings (like a dictionary), BPE allowed \textit{sub-word tokenization} based on arbitrary compression of a text. BPE has two general steps, which are defined differently depending on the training and/or goals of the specific tokenizer:

\begin{enumerate}
\item \textbf{Pre-tokenization:} For the BPE algorithm we need to know what we are compressing \textit{towards}, i.e.\ some sort of idea of what a \textit{token} can be vs. what it can't be. This is defined in the pre-tokenization stage. Typically, pre-tokenization splits on whitespace and punctuation using regular expressions\citep{schmidt2025}. GPT-2 \citep{radford2019}, for example, is a byte-level BPE tokenizer utilizing regular expression pre-tokenization. If we have the input text \textit{``It's always 5 O'clock in New Orleans''} we end up with \texttt{['It', "'s", 'Ġalways', 'Ġ5', 'Ġo', "'", 'clock', 'Ġin', 'ĠNew', 'ĠOrleans']}, where \texttt{'Ġ'} refers to a unique representation for whitespace.

For byte-level tokenizers, they then replace all bytes of the given string with a corresponding representation. This is done by creating a bijective mapping from bytes to some unique character representation. Since there are only 256 unique bytes, this allows an efficient reduction of the vocabulary size while still retaining no unknown tokens during inference, like we saw with the example for the regular expression normalization. It does mean, however, that some characters --- especially those with different script than English --- are represented by several byte-level representations during the pre-tokenization stage. For example, the Chinese character `火' is one character in UTF-8, but is represented by the equivalent string `çģ«' in GPT-2 pre-tokenization. These are the bytes \texttt{C3 A7 C4 A3 C2 AB}.

\item \textbf{Merge:} We define a desired vocabulary size, and perform merges based on the paired frequencies of our current vocabulary applied to our training set corresponding to valid pre-tokenization chunks. For a a byte-level BPE tokenizer our base vocabulary at the start of training will be a few special tokens (like \texttt{<|endoftext|>} $\rightarrow$ 50256) along with all possible bytes (1-256). For a character-level BPE tokenizer it will be the set of unique characters in our training data. For each merge we pick the most frequent pair as our next merge candidate, and then repeat until we reach our desired vocabulary size. For GPT-2 those valid candidates are picked from the regex chunks.

\end{enumerate}

The byte-level tokenization leads to some unintuitive tokens for languages which represents characters as multiple bytes in UTF-8. For example, for the character `火,' GPT-2 tokenizes it as \texttt{['ç', 'ģ«']}, where neither token maps to anything meaningful to humans. \citet{arnett2025, wegmann2025} show the unequal impact of the pre-tokenization decisions in cross-lingual scenarios, where the pre-tokenization can have a detrimental effect on non-Latin languages or languages heavily utilizing diacritics and non-English punctuation rules.

\item[UniGram]
A UniGram language model considers the probability of $z|x,y$ without context, i.e.\ $p(z|x,y) = p(z)$. The probability of a sequence is the product of individual token probabilities. \citet{kudo2018} suggested the utilization of segmentation probability sampling from the UniGram language model. Specifically, in order to act as a regularization method during NMT training. At each step in training we compute a loss over the current vocabulary $V$, and then for each symbol in $V$ we compute how much the loss would change if the symbol was removed. We gradually prune the symbols which would increase the least until we reach a desired vocabulary size. In practice, tokenizers utilizing UniGram, such as SentencePiece \citep{sentencepiece}, approximate this process by utilizing Viterbi decoding to get better performance.

Getting a probability distribution is especially useful in multilingual scenarios, where we might not want to make assumptions about what can and can't constitute a token before we train our tokenizer. \citet{sentencepiece} introduces this framework specifically to be deterministic in decoding and encoding for multilingual scenarios. For example, Japanese doesn't utilize whitespace, while English does. In BPE we make such assumptions during the pre-tokenization, in SentencePiece we don't have to.  However, SentencePiece approaches are not always byte-based, which means unknown tokens are a possibility, depending on the training data. For example, typing an Arabic word `{\arabicfont لغة}' \textit{language} into a monolingual English character-based SentencePiece tokenizer, like the tokenizers in \citet{chang2022}, would end up as \texttt{['<unk>']} (i.e.\ as unknown tokens).

\item[Byte-level Tokenization] Some approaches forego compression, and instead use a mapping from text to their unicode byte representations. The \texttt{By-T5} series of models, introduced by \citet{xue2022}, uses byte tokenization in an attempt to make less assumptions about the input, simplify preprocessing, and help with robustness to unseen tokens. It is similar to the byte-level BPE mapping, but without the regex chunking. For example, the \texttt{By-T5} tokenizer can be replaced wholly by one line of Python code. Let \texttt{text} be a string you want tokenized. 3 denotes the number of reserved special tokens:
\begin{equation}
    \texttt{[int(x) + 3 for x in text.encode("utf-8")]}
\end{equation}

\end{description}

\section{Transformer Architecture}\label{sec:background_transformer}

Most modern large language models are decoder-only models based on the transformer architecture introduced by \citet{vaswani2017}. At a high level, a decoder-only transformer processes a sequence of token representations and builds contextualized embeddings through a series of layered transformations. In each layer, token representations are gradually updated by incorporating information from other tokens in the sequence. The resulting representations are then used to predict the next token in the sequence.

Before these transformations can take place, the discrete tokens produced by the tokenizer must be converted into vector representations that the model can process. This is done using an input embedding layer, which maps each token in the vocabulary to a dense vector. Since the transformer architecture does not inherently encode sequence order, positional information is added to these embeddings to allow the model to distinguish between different token positions in the input. The resulting representations (token embeddings + positional encodings) are then passed through a stack of transformer blocks, which iteratively refine the representations. After the final layer, a normalization step and a linear projection map the representations to the vocabulary space, producing a probability distribution over possible next tokens. This high-level overview is pictured in Figure~\ref{fig:transformer_layers}.

Transformer blocks consist of a self-attention layer and a feed-forward network, each wrapped with residual connections and layer normalization, visualized in Figure~\ref{fig:transformer_block}. The central concept in the transformer block is the self-attention layer. In a self-attention layer, three linear projections of an input sequence with dimensionality $d$ produces the \textit{Query}, \textit{Key}, and \textit{Value} vectors. These vectors $Q$, $K$, and $V$ are then used to calculate attention scores:


\begin{equation}\label{eq:attention}
\text{Att}(Q, K, V) = \text{softmax} \left( \frac{QK^{\top}}{\sqrt{d}} \right) V
\end{equation}

\noindent
with softmax being defined as

\begin{equation}\label{eq:softmax}
\text{softmax}(z_i) = \frac{e^{z_i}}{\sum_{j=1}^{k} e^{z_j}}
\end{equation}

\noindent
where $z = (z_1, \dots, z_k)$ is a vector of input scores, and as output gives a normalized probability distribution from every token to every token summing to $1$. As noted by \citet{correia2019} this normalization makes it possible, in principle, for tokens to incorporate information from all other tokens in the sequence. In decoder-only transformers, training is performed using next-token prediction, where attention scores for tokens occurring after the current token are masked. To allow the model to capture different types of relationships in parallel, transformers use \textit{multi-head attention}. This involves applying Equation \ref{eq:attention} many times in parallel, effectively forming several \textit{attention heads} which can attend to distinct parts of the sequence. Each head is calculated as such, where $W_i$ represents the learnable weight matrices for the $i$-th attention head:

\begin{equation}\label{eq:multihead}
\text{head}_i (Q, K, V) = \text{Att}(QW_i^{Q}, KW_i^{K}, VW_i^{V})
\end{equation}

Many modern LLMs, such as Llama2 \citep{llama2} and Qwen \citep{qwen2.5,qwen2}, utilize Grouped Query Attention \citep{ainslie2023}, where multiple heads share keys and values. This leads to faster training and inference, at the potential cost of increased interactions between heads compared to standard multi-head attention.

\section{Mechanistic Interpretability}\label{sec:background_interpretability}

Mechanistic interpretability is the field concerned with understanding the inner workings of neural networks. Early efforts were mostly concerned with vision \citep{cammarata2020a}, and focused on identifying humanly interpretable features. For example, features were found in neural models corresponding to grocery store detectors \citep{nguyen2016}, curve detectors \citep{cammarata2020b}, or even multi-modal features \citep{goh2021}.

Unlike vision, where we can easily \textit{visualize} our features, text-based LLMs don't offer the same direct type of interpretability. \citet{elhage2022a} argues this is because of the superposition hypothesis, where a model stores more features than it directly has dimensionality for. It does this by utilizing \textit{poly-semantic} neurons, which are neurons which activate for many seemingly unrelated things and are only disambiguated to its \textit{mono-semantic} variant by combined linear activation with other neurons. \citet{elhage2022b} demonstrated superposition emerging during training of small toy example language models, and that as feature sparsity increases, so does the prevalence of neurons stored in superposition. \citet{bricken2023} successfully derived mono-semantic features using a sparse auto-encoder, a type of sparse dictionary learning.

\subsection{Transformed Language Models}

For autoregressive decoder transformer models, like GPT, we solely do next-token prediction which we feed back into the model. We process one token at a time, perform some transformation to our current sequence, and get a next-token prediction, which we append to the current sequence. At each layer we have a self-attention mechanism and a Feed Forward Network (FFN) which updates these token representations. Each update is added to the output from the previous, forming the \textit{residual stream}. At the end, we soft-max to our vocabulary dimension to get a probability distribution for the next token. At each given layer our representation thus represents the residual stream up until that layer.

We can use this to probe the models internal thinking. One popular method is the \textit{logit lens} \citep{nostalgebraist}, which decodes the next token prediction from the output of each hidden state. \citet{ghandeharioun2024} introduced \texttt{PatchScopes}, which first runs an inference run on a piece of text and saves the output of each hidden state from each layer. We then ``patch'' a new inference call with those outputs. For example with a prompt like \texttt{`X, X, X, '} where we replace \texttt{X} with the hidden state output, and we then measure if it successfully repeats the input just from the hidden layer outputs. Thus enables more flexibility due to multi-token probing.

\subsection{Detokenization}

\citet{elhage2022a} introduced the notion of \textit{detokenization}, the hypothesis that LLMs use the early layers to map subword tokens to more useful concepts that the model can then use throughout the middle layers, and before \textit{retokenization} at late layers which is the opposite process. \citet{feucht2024} calls this the \textit{implicit} vocabulary of the model. They train linear probes for each layer in a Llama 2 7B / Llama 3 8B to show that multi-token entities, words, and phrases go through ``erasure,'' wherein information about tokens is primarily focused in the early layers of the model.

The parallel multi-head structure allows attention heads to become specialized for specific tasks. \citet{liu2025_language_attention_heads} identify language-specific and language-agnostic attention heads, which enables them to guide the model towards specific languages in generation. \citet{hanna2023} identified a circuit in GPT2 responsible for the `XX greater than YY' relation, and found that at the last token position a few in-circuit attention heads attend primarily to the YY position. They check this using the logit lens approach \citep{nostalgebraist}, which lets them project the attention head to vocabulary space and get output token prediction. It shows one head up-weights the YY token prediction over any other token. \citet{carrasco2024} found a circuit of 8 attention heads responsible for three-letter acronym detection in GPT2, which they tested using activation patching, which involves replacing layer activation values with a corrupted alternative. \citet{voita2019} find that only a small subset of attention heads are crucial for translation performance, as well as identifying \textit{patterns} in attention heads.
